## 📌 篇一 · 痛点叙事流（主推 · 适合首发）

### 标题（3 选 1）

1. **vibe coding 三个月，我变成了 docs 文件的奴隶 😮‍💨**（情绪+反差，首选）
2. **打开 AI 写的 docs 文件夹，我自己都看不懂了**（共鸣+悬念）
3. **AI 写得越快，我越搞不清自己在做啥**（金句+痛点）

> 建议首发用 1。emoji 放结尾，标题控制在 18 个字以内（小红书移动端会截）。

### 封面文字脚本（用 Canva / 醒图做图）

**主标语**（占据上半屏幕，加粗）：
> 和 AI 聊了 3 个月
> 我桌面多了 200 个文档

**副标语**（小字，深灰色）：
> vibe coder 的真实日常

**配图建议**：
- 上半部分：真实截屏一张混乱的 docs 文件夹（建议自己截一张，越乱越真实）
- 下半部分：一个崩溃表情的真人头像 / 表情包
- **颜色**：避免亮蓝色（会被识别为广告），用米黄/灰白底+黑字最自然
- **不要加边框、不要加 logo、不要加品牌色**——越像随手截图越好

### 正文

```
家人们，我必须吐槽一下用 AI 写代码的副作用——

不是工资变少，
也不是头发变多，
是我桌面的文档变得没人看得懂 🙃

跟 Claude / Cursor 聊一下午，
回头打开 docs 目录——

✅ PRD 改过三版，三个文件名都不一样
✅ 各种 plan 写了七八个，没人回头看
✅ "先记一下回头再整理"的便签：23 个
✅ AI 生成的接口文档：每次格式都不一样
✅ 最离谱的：我自己都讲不清这个项目做到哪了

试过 Notion？要复制粘贴一遍。
试过 Jira？太重。
试过自己手动整理？整理完发现 AI 又生成了 5 个新的 🫠

后来我索性自己写了个开源小工具——
扫一遍文档目录，自动生成一个看板，
浏览器双击打开就能看到：
🎯 哪些模块在做、哪些在等、哪些已完成
📊 每个模块的进度条
📅 各阶段的时间线
✏️ 点开看板每个卡片，还能直接看到关联的设计稿和方案

最香的是 —— 它会逼着 AI 按统一规范写文档。
AI 写错了直接报错，不再每次格式都不一样。

自用一个多月，
我又开始能讲清楚自己在做啥了 🥹

完全免费开源，
项目名叫 docs-cockpit（驾驶舱的意思），
感兴趣可以去 GitHub 搜这个名字。

#AI编程 #程序员日常 #独立开发者 #Claude #数字工具
#效率方法 #工作流 #开源工具 #副业项目 #ai工具推荐
```

### ⚠️ 注意

## 📌 篇二 · 数字反差流（适合发完篇一 3-5 天后再发）

### 标题（3 选 1）

1. **我让 AI 自己收拾它写的 200 个文档，结果……**（悬念，首选）
2. **AI 写文档很爽，但我已经一个月没读完过了**（反差自嘲）
3. **测试了 3 个工具，终于找到管 AI 文档的方法**（评测向）

### 封面文字脚本

**对比版**：
> 左半：一堆乱七八糟 md 文件截图 + 文字「以前」
> 右半：dashboard 看板截图 + 文字「现在」
> 中间：箭头 →

**或：单图版**（更省事）：
> 截一张你 docs-cockpit 跑出来的 dashboard 截图
> 顶部加文字：**「文档全自动变看板✨」**
> 角落加小字：**「真·AI 协作工程化」**

### 正文

```
🙋‍♀️ 上次发了一篇吐槽 AI 写文档太多管不过来，
评论区好几个姐妹问我后来咋解决的——
今天来交个作业 ✍️

先说结论：
不需要换工具、不需要付费 SaaS、
只需要给 AI 一份"写文档的规范"，
然后让规范自己监督 AI。

具体怎么做：

① 给所有项目文档加一个统一的"前置信息"
比如每个文档开头几行写清楚：
  - 这是哪个模块（id）
  - 当前状态（进行中 / 已完成 / 已阻塞）
  - 进度百分比
  - 关联到哪个需求 / 设计稿
（这一步是关键，AI 不再发明规则，只能照填）

② 装一个小工具自动校验
AI 偶尔会偷懒——说"已完成"但进度只填 60%。
工具会立刻报错指出来：
"⚠️ 这里状态和进度对不上"
省得上线那天才发现 🤡

③ 一行命令把整个文档目录变看板
看板长这样👇（图 2/图 3）
点每个卡片能直接看关联的设计稿和方案
本地浏览器打开，不联网不上传

——

整体流程跑通后，我发现最大的改变不是工具好用，
是 AI 终于变得"可控"了：
它不再每次创造新规则，
而是在我给的框架里干活。

这套东西我打包成了一个开源项目，
叫 docs-cockpit，
不收费、MIT 协议，谁用谁知道。
（感兴趣可以去 GitHub 搜，名字一搜就到）

下条想分享：怎么让 AI 帮我自动写周报 ✋

#AI工具 #程序员 #独立开发者 #效率工具
#vibecoding #ChatGPT #Claude #数字游民 #开源 #工作流
```

### ⚠️ 注意

- 开头"上次发了一篇"——和篇一形成系列，平台会判定为"账号活跃"
- 写"GitHub 搜"是允许的，写"GitHub 链接"是限流的
- 三步骤结构是小红书最受欢迎的格式（"教程"标签流量大）

## 📌 篇三 · 工具控/科普流（适合养号一段时间后做"专业感"内容）

### 标题（3 选 1）

1. **AI 时代最大的 BUG 不是大模型，是文档失控**（金句+反共识，首选）
2. **为什么用 AI 越多，你越搞不清自己在做啥**（戳痛点）
3. **大模型不可靠？给它一个规范就行了**（反直觉）

### 封面文字脚本

**金句封面**（极简风）：
> 黑底白字，居中一行大字：
> 「AI 时代最大的 BUG，
>   不是大模型不准，
>   是没人给它定规范。」
>
> 底部一行小字：
> 「— 一个 vibe coder 的反思」

**或**：手写笔记风（更亲和）：
> 在 iPad 备忘录 / GoodNotes 里手写：
> 「① 大模型每次格式都不一样
>   ② 文档越多越没人看
>   ③ 项目状态越来越糊
>   ↓
>   解法：给 AI 一份强 schema」
> 截屏当封面

### 正文

```
认真聊一个 vibe coder 圈子里没人讨论的问题：

不是"AI 会不会替代程序员"，
不是"Cursor vs Claude Code"，
是—— **大模型生产的文档，正在失控**。

🧠 我观察：

用 AI 写代码 3 个月以上的人，
项目文档量会暴涨 5-10 倍，
但能用上的不到 20%。

为什么？

因为大模型有一个被严重低估的缺陷：
**它没有"项目记忆"，每次都是从零开始**。

你问它"plan 文档应该有哪些字段"
今天它给 A 答案，明天给 B 答案，
后天写 RFC 文档又用 C 格式。

久而久之，
一个项目里的 200 个文档，
没有任何两个格式一致。
肉眼根本读不懂自己的项目状态。

🛠 真正的解法只有一个：

把"规范"从 AI 的脑子里拿出来，
变成一份**可校验、可强制、机器可读**的文件，
让 AI 每次写文档前先读它，
写完后还要过验证。

具体到我这套实践：
- 一份 schema 定义所有字段
- 一个 skill 告诉 AI 怎么写
- 一个 validator 写完即查
- 一个 dashboard 把所有文档可视化

跑通之后，
AI 还是那个 AI，
但项目终于"可看、可问、可推进"了。

—— 顺手把这套东西做成了开源项目，
叫 docs-cockpit，MIT 协议，免费用。
（搜名字就能找到，不在这放链接了）

如果你也是 vibe coder，
建议想想这个问题：
**你今天的 docs，半年后还看得懂吗？**

#AI编程 #程序员日常 #独立开发者
#知识管理 #效率方法 #工具控
#claudecode #cursor #ai工具 #思考
```

### ⚠️ 注意

- "认真聊一个"开头适合"专业感"账号定位，会吸引技术博主转发
- 全篇没有任何"求点赞/求收藏/求私信"——反而更容易被转
- 最后一句"半年后还看得懂吗？"是引发互动的开放式问题
- 标签里"#思考""#知识管理"会蹭非技术圈流量

### 置顶评论

```
补一个反共识观点：

很多人以为"大模型不可靠"是模型本身的问题，
其实是我们给它的「上下文系统」太弱。

同一个 Claude，
你给它一份 schema + validator，
它就突然"靠谱"起来了——
不是它变聪明了，是它的环境变规范了。

这是 vibe coding 工程化的下一站 🪐
```

## 🎯 三篇组合发布节奏

| 顺序 | 篇目 | 发布时机 | 目标 |
|---|---|---|---|
| 1 | 篇一（痛点叙事） | 工作日 12:00 或 21:30 | 触达共鸣，做基础流量池 |
| 2 | 篇二（数字反差） | 篇一发完 3-5 天后，工作日 12:00 | 蹭篇一的复访流量，做转化 |
| 3 | 篇三（工具控/科普） | 篇二发完 5-7 天后，工作日 21:00 | 打专业感，吸引 KOL 二次传播 |

**任何一篇阅读量 < 200** → 立刻删，避免账号权重持续下跌
**任何一篇阅读量 > 1000** → 评论区高频互动 24h（前 6 条评论必回）
**累计 3 篇都没起量** → 大概率是账号底子被 v1 限流污染了，建议换新号重发

## 📦 配合落地页（site/index.html）

发笔记的同时，**确保 GitHub Pages 已部署**：
```
https://guohao1020.github.io/docs-cockpit/
```

当用户从小红书搜到项目名 → 进 GitHub → 落地页 → README → 安装。
这条转化链条上，落地页是关键节点，因为：
- GitHub README 给开发者看（信息密度高，新人会逃）
- 落地页是中文营销页（H1 戳痛点，FAQ 解疑，CTA 分人群）

笔记里**不要直接挂落地页链接**——让用户自然搜索到。
