# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

`docs-cockpit` is an **open-source MIT-licensed** project that turns YAML-frontmatter-driven markdown into a single-file project Kanban dashboard. It ships as **two artifacts from the same repo**:

1. **A Python CLI** (`docs-cockpit` console script · `pyproject.toml::project.scripts`) installable via pip / uv tool / pipx
2. **A Claude Code plugin** (`.claude-plugin/plugin.json` + `skills/` + `commands/`) installable via `/plugin install docs-cockpit@docs-cockpit`

The plugin's skills reach into the CLI for the actual MD → HTML build — they're not duplicates of each other. The CLI is the runtime; the plugin is the interface layer for AI agents.

Because the project is open-source and used in production on multiple downstream projects (e.g. `D:/harvey_work/Sourcery`, `D:/shulex_work/bastion`), changes here surface as user-visible plugin updates. Treat any change to `skills/*/SKILL.md`, the validator, or the HTML template as a release event — bump the version per the SemVer convention below.

## Common commands


The two reference projects used for end-to-end smoke testing are `D:/harvey_work/Sourcery` (24 modules, 11 concepts) and `D:/shulex_work/bastion` (49 modules). Build them after substantive `build.py` / template changes to verify nothing regressed.

### "Lint" in this project

`docs-cockpit lint` is **frontmatter validation against the canonical schema in `docs-cockpit-author`** — NOT Python code linting. There is no ruff / mypy / black configured for this repo's own Python source. If you want code-level linting, add it and document it.

### Release / version bump

A release touches **four files together**:

```
docs_cockpit/__init__.py          # __version__ = "X.Y.Z"
.claude-plugin/plugin.json        # "version": "X.Y.Z"
.claude-plugin/marketplace.json   # "version": "X.Y.Z"
CHANGELOG.md                      # add ## [X.Y.Z] · YYYY-MM-DD section above prior entry
```

After commit + push, users on the plugin can pull the update via `docs-cockpit upgrade` (handles plugin-cache clear + atomic restart prompt).

## SemVer convention

This project deviates from strict SemVer to encode the user-visible blast radius:

- **patch** (`0.7.1`, `0.7.2`) — CLI-only / template-only fix. No SKILL.md changes. Plugin can be updated without restarting Claude Code (just the CLI).
- **minor** (`0.8.0`, `0.9.0`, `0.10.0`) — new feature, new schema field, new CLI subcommand, new skill, or any SKILL.md change. Plugin cache MUST be cleared + Claude Code restarted (the `upgrade` CLI handles this atomically).
- **major** — config schema break (`docs-cockpit.yaml` shape changes incompatibly). None yet.

A change to `skills/*/SKILL.md` is by definition at least minor — those files are loaded into Claude's context, and stale ones cause hard-to-debug routing bugs.

## §0 · driver-seat 角色定位(2026-05-18 round-4 重新框架)

**关键 architectural shift**:driver-seat **不是自给自足的精度引擎** · 它是 **AI 工具(Claude Code / Codex / Cursor)的副驾**。

| 维度 | 旧定位(误读) | 新定位(正确) |
|---|---|---|
| 精度来源 | docs-cockpit 用 python regex / 反向 index 算 | **LLM 来做语义精度** · python 只做解析层(anchor 语法 / 行号 / heading slug) |
| 核心问题(round-3 提的 doc anchor / module-aware highlight) | 怎么用 python 判断 subtask 跟 doc 哪段相关 | **怎么教 AI** 在写 module MD 时直接产出精准 anchor · 或 **让 AI 检查现有 module** 输出 refinement patch |
| docs-cockpit 的核心责任 | 算精度 + 渲染 | **上下文供给(context)+ UI 展示 + 工作流编排** |

**implications**:
- v0.11 alpha.6(split-view UI)· python 仅算解析层精度 · 让 AI 输出的 anchor 能渲染
- v0.11 alpha.7 · **AI-augmented precision 两条线**:
  - 模式 3:升级 `docs-cockpit-author` SKILL.md · 教 Claude 写 module MD 时直接产出精准 anchor
  - 模式 2:split-view 加「🤖 Ask AI to refine this module」按钮 · 走 prompts.js 延伸 · AI 检查现有 module + 输出 YAML patch
- v0.12 · 模式 1 · Claude API build-time 自动 augment + MCP server 直连(去掉 copy-paste)

详见 sub-plan:`docs/plans/P-v0.11-ai-augmented-precision-alpha7-2026-05-18.md`

## §1 · Problem Statement

docs-cockpit 现状是一个能渲染 module/concept/sprint/progress 的 **可视化看板**,但用户体验暴露出 5 个核心缺口:

1. **模块-任务-子任务-spec/plan/RFC 关联不到位** · subtasks 目前是 `list[str]` · 没有 ID · 没法跨 build 稳定追踪 · 没法跟具体 spec/plan/RFC 双向锚定
2. **子任务无法直接定位到代码位置** · 用户写"补 BrowserVendor abstraction"但 cockpit 不知道这指的是 `sourcery/worker/browser_vendor.py:42-89`
3. **任务清单状态控制不闭环** · localStorage 已经能勾选,但状态不回流到 MD · 长期会跟 source-of-truth 漂移
4. **用户文档质量参差** · 新用户经常 frontmatter 残缺 / subtasks 是 stub · cockpit 没法引导他们写出合规的 source MD
5. **驾驶舱不能执行任务** · 用户看到一个 in-progress subtask · 还得手动去拼"我现在要做什么 / 哪些文件 / 哪些 spec / 哪些前置" · cockpit 应该一键给出可执行的 prompt

这 5 个缺口本质上是同一件事:**cockpit 还停留在「展示」· 没有进入「驱动」**。

## §2 · What Makes This Cool

把 cockpit 从「项目状态展示器」升级为「AI 协作驾驶舱」。落地后的「驾驶员体验」是:

- 用户打开 cockpit · 看到 6 个 in-progress module · 每个 module drawer 里有真实子任务 checklist
- 点开任意 subtask · 看到 `code:` anchor 直链到 `sourcery/worker/browser_vendor.py:42-89` 的对应行 · 旁边显示关联的 spec §2.1 摘要
- 点「复制提示词」· 拼好的 prompt 进剪贴板 · 包含:module spec § / subtask 详情 / linked docs 摘要 / 代码 anchor 内容 / 当前 status
- 用户粘到 Claude 跑 · Claude 知道现在要做什么 / 在哪里改 / 验收标准是什么
- 跑完 · Claude 输出 frontmatter patch · 用户审阅后落回 MD · 状态闭环

这把 docs-cockpit 从「写好文档之后可视化用」升级到「跟 AI 一起做事的指令中心」。

## §3 · Constraints

- **MIT 开源 · 不能 vendor lock** · prompt template 必须用通用 Jinja2 · 不绑死 Claude / Anthropic SDK
- **下游 Sourcery + bastion 已经在生产用 v0.10** · v0.11 必须向后兼容旧 `subtasks: list[str]` 格式
- **Plugin 不能 pip-install** · CLI 升级走 `docs-cockpit upgrade` · plugin marketplace cache 必须随 minor bump 失效
- **HTML template 是 single-file `__DOCS_JSON__` 替换** · 不能引入运行时 JS framework · prompt 必须在 build 时渲染为静态 string 嵌入
- **不能侵入式改用户 MD** · LLM 改写一律走「生成 prompt 让用户跑 Claude」路径 · MD 永远是 SSOT
- **CHANGELOG / SemVer 偏移**:任何 SKILL.md 变更都是 minor · 这次同时改 `docs-cockpit-author` / `docs-cockpit` 两个 skill + build.py · 必须 0.11.0 minor

## §4 · Premises (locked in Phase 3)

1. **方向 premise** · v0.11+ 不是更精致的看板 · 是从「可视化看板」升级到「AI 协作驾驶舱」
2. **Schema 演进 premise** · `subtasks` 从字符串 list 升级为对象数组 · 字段至少 `id / title / status / code? / docs?` · **minor · 完全向后兼容旧 string 格式**(validator 把 string 当 `title` · 自动 normalize · 不报 error)
3. **Prompt scaffolding 边界 premise** · v1 面向人 copy-paste · 不直连 Claude · MCP / plugin hook 是 v2 范围
4. **Trust premise** · 「自动优化文档」不能侵入式改用户 MD · MVP 走「生成 prompt 让用户跑」· 保留 MD 是 SSOT 的信任锚
5. **Hard gate premise** · 本轮 office-hours 只设计 · 不实施 · 实施由后续 task 触发(/plan-eng-review 或直接动手)
