## §0 · driver-seat 角色定位(2026-05-18 round-4 重新框架)

**关键 architectural shift**:driver-seat **不是自给自足的精度引擎** · 它是 **AI 工具(Claude Code / Codex / Cursor)的副驾**。

| 维度 | 旧定位(误读) | 新定位(正确) |
|---|---|---|
| 精度来源 | docs-cockpit 用 python regex / 反向 index 算 | **LLM 来做语义精度** · python 只做解析层(anchor 语法 / 行号 / heading slug) |
| 核心问题(round-3 提的 doc anchor / module-aware highlight) | 怎么用 python 判断 subtask 跟 doc 哪段相关 | **怎么教 AI** 在写 module MD 时直接产出精准 anchor · 或 **让 AI 检查现有 module** 输出 refinement patch |
| docs-cockpit 的核心责任 | 算精度 + 渲染 | **上下文供给(context)+ UI 展示 + 工作流编排** |

**implications**:
- v0.11 alpha.6(split-view UI)· python 仅算解析层精度 · 让 AI 输出的 anchor 能渲染
- v0.11 alpha.7 · **AI-augmented precision 两条线**:
  - 模式 3:升级 `docs-cockpit-author` SKILL.md · 教 Claude 写 module MD 时直接产出精准 anchor
  - 模式 2:split-view 加「🤖 Ask AI to refine this module」按钮 · 走 prompts.js 延伸 · AI 检查现有 module + 输出 YAML patch
- v0.12 · 模式 1 · Claude API build-time 自动 augment + MCP server 直连(去掉 copy-paste)

详见 sub-plan:`docs/plans/P-v0.11-ai-augmented-precision-alpha7-2026-05-18.md`

## §1 · Problem Statement

docs-cockpit 现状是一个能渲染 module/concept/sprint/progress 的 **可视化看板**,但用户体验暴露出 5 个核心缺口:

1. **模块-任务-子任务-spec/plan/RFC 关联不到位** · subtasks 目前是 `list[str]` · 没有 ID · 没法跨 build 稳定追踪 · 没法跟具体 spec/plan/RFC 双向锚定
2. **子任务无法直接定位到代码位置** · 用户写"补 BrowserVendor abstraction"但 cockpit 不知道这指的是 `sourcery/worker/browser_vendor.py:42-89`
3. **任务清单状态控制不闭环** · localStorage 已经能勾选,但状态不回流到 MD · 长期会跟 source-of-truth 漂移
4. **用户文档质量参差** · 新用户经常 frontmatter 残缺 / subtasks 是 stub · cockpit 没法引导他们写出合规的 source MD
5. **驾驶舱不能执行任务** · 用户看到一个 in-progress subtask · 还得手动去拼"我现在要做什么 / 哪些文件 / 哪些 spec / 哪些前置" · cockpit 应该一键给出可执行的 prompt

这 5 个缺口本质上是同一件事:**cockpit 还停留在「展示」· 没有进入「驱动」**。

## §2 · What Makes This Cool

把 cockpit 从「项目状态展示器」升级为「AI 协作驾驶舱」。落地后的「驾驶员体验」是:

- 用户打开 cockpit · 看到 6 个 in-progress module · 每个 module drawer 里有真实子任务 checklist
- 点开任意 subtask · 看到 `code:` anchor 直链到 `sourcery/worker/browser_vendor.py:42-89` 的对应行 · 旁边显示关联的 spec §2.1 摘要
- 点「复制提示词」· 拼好的 prompt 进剪贴板 · 包含:module spec § / subtask 详情 / linked docs 摘要 / 代码 anchor 内容 / 当前 status
- 用户粘到 Claude 跑 · Claude 知道现在要做什么 / 在哪里改 / 验收标准是什么
- 跑完 · Claude 输出 frontmatter patch · 用户审阅后落回 MD · 状态闭环

这把 docs-cockpit 从「写好文档之后可视化用」升级到「跟 AI 一起做事的指令中心」。

## §3 · Constraints

- **MIT 开源 · 不能 vendor lock** · prompt template 必须用通用 Jinja2 · 不绑死 Claude / Anthropic SDK
- **下游 Sourcery + bastion 已经在生产用 v0.10** · v0.11 必须向后兼容旧 `subtasks: list[str]` 格式
- **Plugin 不能 pip-install** · CLI 升级走 `docs-cockpit upgrade` · plugin marketplace cache 必须随 minor bump 失效
- **HTML template 是 single-file `__DOCS_JSON__` 替换** · 不能引入运行时 JS framework · prompt 必须在 build 时渲染为静态 string 嵌入
- **不能侵入式改用户 MD** · LLM 改写一律走「生成 prompt 让用户跑 Claude」路径 · MD 永远是 SSOT
- **CHANGELOG / SemVer 偏移**:任何 SKILL.md 变更都是 minor · 这次同时改 `docs-cockpit-author` / `docs-cockpit` 两个 skill + build.py · 必须 0.11.0 minor

## §4 · Premises (locked in Phase 3)

**§11 · 写 module MD 时的 AI 工作流**

教 Claude 收到「帮我写 M07 module」请求时的标准流程:

1. **先读相关 plan / RFC body**(用户引用的 sprint plan / driver-seat plan / module-related RFC)
   - 不是 grep · 是真读 · 看每个 section 在讲什么
2. **跨参考拆 subtask**:
   - 每个 subtask 应该对应 plan 的某个具体 section / RFC 的某个决策
   - subtask.title 用 plan 里的术语 · 保语义一致
3. **填精准 `code:` anchor**:
   - 阅读 repo · 找 subtask 实际要改的文件 + 函数 + 行号
   - 输出 `code: docs_cockpit/build.py:42-89` 这种精确格式 · 不是 `docs_cockpit/`
4. **填精准 `docs:` anchor**:
   - 用 `docs/plans/x.md#§6.1` 或 `:120-180` 锚到 plan 具体 section / 行号
   - 不要只写 `docs/plans/x.md`(整个 doc · 用户看不到重点)
5. **跨 subtask 一致性 self-check**:
   - 同 module 内 subtask 之间是否有依赖 · 在 title 里 hint(「Lane A 完成后做 Lane B」)
   - 同一 plan section 被多 subtask 引用 · 是否过度细分(merge 一下)

**§12 · 跨 module / doc 一致性 self-check**

教 Claude 在产出 module 后做一次 cross-check:

1. **doc backref**:同一 plan / RFC 被多 module 引用 · 检查 anchor 是否互不重复(M01 §6.1 / M02 §6.2 互斥而不是都指 §6.1)
