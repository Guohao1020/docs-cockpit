# Sprint 0.19 plan

## §1 · 用户得到什么(详写 goals)

**G1 · sprint-plan 一等公民**:用户写一个 `docs/plans/V0.20.md` · docs-cockpit 自动识别 type=sprint-plan · 跑 schema 校验 · dashboard 后续(v0.19.3)会渲染。版本视角第一次有 single source of truth · 不靠人脑串。

**G2 · DoR 自动校验**:用户原话「每个版本开始前需要校验需求对齐 + LLM 参考文档充分性」· lint_sprint_readiness 对每个 status=planned/in-progress 的 sprint-plan 自动跑:每个 in_scope subtask 必须 trace 到 prd_refs · 必须有 @code/@docs anchor。漏一条 warn 一次。

**G3 · 三个 CLI subcommand**:`sprint init` scaffold 新 sprint(自动从 state.json 反查 module.sprint 字段填 in_scope[])· `sprint check` 跑 DoR · `sprint list` 看全景。

**G4 · 跟现有 lint pipeline 集成**:`docs-cockpit build` / `docs-cockpit lint` 自动跑 sprint-readiness。默认 opt-in · 没 sprint-plan 不报警 · 不破坏没用 sprint workflow 的老项目。

## §2 · 跟上 sprint 的衔接

v0.18 sprint plan 把 MCP / lint / body patch 三件 P0 gap 收口 · v0.19 在那个 lint 框架基础上新增 sprint-readiness category · 走同款 Issue.category 机制 · 复用 `--include` / `--exclude` filter 接口。

## §3 · 风险 + 假设

- **风险 · scope creep**:sprint plan 这事容易过设计 · 比如想搞 LLM-driven composer / dashboard timeline 重做 · 全塞 v0.19.0 会爆。✅ mitigation:out_of_scope[] 已显式 freeze v0.19.x 后续工作。
- **风险 · 老 project 误报**:enforce_sprint_plans 默认 false · 没 sprint-plan 不报警。✅
- **假设**:用户已经在用 `sprint:` 字段命名 sprint(Sourcery 跑 M1.1 / M1.5.a 那套)· 新 sprint-plan 跟这套兼容 · `_normalize_sprint_id` 自动接 `V0.19` / `0.19` / `M1.5.a` 等多种写法。

## §4 · 验证 · 谁来跑 demo

- **G1 demo**:`docs-cockpit build` 看到 type=sprint-plan 文件走 validate_sprint_plan · 缺字段报 error。dogfood 本 sprint 的 V0.19.md。
- **G2 demo**:dogfood 跑 `docs-cockpit sprint check 0.19` · 应该输出 0 issue(自家 module 已经全规范)。
- **G3 demo**:`docs-cockpit sprint list` 表格 · 看 V0.18 done · V0.19 in-progress。
- **G4 demo**:`docs-cockpit lint` 自动 include sprint-readiness。
