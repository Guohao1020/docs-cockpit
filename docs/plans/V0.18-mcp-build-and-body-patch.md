# Sprint 0.18 plan

## §1 · 用户得到什么(详写 goals)

**G1 · 副驾 MCP 闭环**:Claude Code / Cursor 改完 MD 后不用切回 CLI 跑 `docs-cockpit build`。直接 MCP 调 `cockpit_build` tool · 等价的事在 server side 跑完 · 同步返回 issue 摘要 · 副驾接着读 `cockpit://state` 拿 fresh data。这是 driver-seat mode 1 一步收口动作。

**G2 · body checklist 行级 edit**:用户 module MD body checklist 行(plan §6.1 推荐姿势)inline `@code:`/`@docs:` annotation 现在能走 MCP 精确 add / replace / remove。verify CLI 输出的 ❌ wrong verdict 现在能自动落地 · 不用手工 sed。

**G3 · lint = build**:跟 0.17 之前「lint 不查 anchor / build 查 anchor」mental model 不一致告别。`docs-cockpit lint` 现在跑跟 build 完全相同的 issue collection · 加 `--include` / `--exclude` / `--legacy-schema-only` 给配置弹性。

## §2 · 跟上 sprint 的衔接

上 sprint(v0.17)新增 `lint_subtask_anchors` + `docs-cockpit verify` CLI · v0.18 拿用户 dogfood 反馈三个 gap 续上 · 把 0.17 的 lint / verify 跟 MCP / CLI 全连成一气。

## §3 · 风险 + 假设

- **风险 · half-break**:CI 配 `lint --strict-warn` 项目可能突然挂 → 加 `--legacy-schema-only` 兜底。✅ 已 mitigate。
- **假设**:MCP `mcp` Python SDK 包 stable · 新 tool registration 不会跟老 client breaking。✅ 实测 ok。

## §4 · 验证 · 谁来跑 demo

- **G1 demo**:in-process smoke test · 跑一遍 `_handle_cockpit_build` · 看返回 JSON 干净不含 build noise。✅ 通过。
- **G2 demo**:body_patch 5 场景 unit test · 包括 conflict + idempotent。✅ 通过。
- **G3 demo**:同一个 Sourcery commit 跑 `docs-cockpit lint` 跟 `docs-cockpit build` · issue count 一致。✅ 通过。
