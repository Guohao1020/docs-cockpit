## §1 · Problem Statement

docs-cockpit 现状是一个能渲染 module/concept/sprint/progress 的 **可视化看板**,但用户体验暴露出 5 个核心缺口:

1. **模块-任务-子任务-spec/plan/RFC 关联不到位** · subtasks 目前是 `list[str]` · 没有 ID · 没法跨 build 稳定追踪 · 没法跟具体 spec/plan/RFC 双向锚定
2. **子任务无法直接定位到代码位置** · 用户写"补 BrowserVendor abstraction"但 cockpit 不知道这指的是 `sourcery/worker/browser_vendor.py:42-89`
3. **任务清单状态控制不闭环** · localStorage 已经能勾选,但状态不回流到 MD · 长期会跟 source-of-truth 漂移
4. **用户文档质量参差** · 新用户经常 frontmatter 残缺 / subtasks 是 stub · cockpit 没法引导他们写出合规的 source MD
5. **驾驶舱不能执行任务** · 用户看到一个 in-progress subtask · 还得手动去拼"我现在要做什么 / 哪些文件 / 哪些 spec / 哪些前置" · cockpit 应该一键给出可执行的 prompt

这 5 个缺口本质上是同一件事:**cockpit 还停留在「展示」· 没有进入「驱动」**。

## §2 · What Makes This Cool

把 cockpit 从「项目状态展示器」升级为「AI 协作驾驶舱」。落地后的「驾驶员体验」是:

- 用户打开 cockpit · 看到 6 个 in-progress module · 每个 module drawer 里有真实子任务 checklist
- 点开任意 subtask · 看到 `code:` anchor 直链到 `sourcery/worker/browser_vendor.py:42-89` 的对应行 · 旁边显示关联的 spec §2.1 摘要
- 点「复制提示词」· 拼好的 prompt 进剪贴板 · 包含:module spec § / subtask 详情 / linked docs 摘要 / 代码 anchor 内容 / 当前 status
- 用户粘到 Claude 跑 · Claude 知道现在要做什么 / 在哪里改 / 验收标准是什么
- 跑完 · Claude 输出 frontmatter patch · 用户审阅后落回 MD · 状态闭环

这把 docs-cockpit 从「写好文档之后可视化用」升级到「跟 AI 一起做事的指令中心」。

## §3 · Constraints

- **MIT 开源 · 不能 vendor lock** · prompt template 必须用通用 Jinja2 · 不绑死 Claude / Anthropic SDK
- **下游 Sourcery + bastion 已经在生产用 v0.10** · v0.11 必须向后兼容旧 `subtasks: list[str]` 格式
- **Plugin 不能 pip-install** · CLI 升级走 `docs-cockpit upgrade` · plugin marketplace cache 必须随 minor bump 失效
- **HTML template 是 single-file `__DOCS_JSON__` 替换** · 不能引入运行时 JS framework · prompt 必须在 build 时渲染为静态 string 嵌入
- **不能侵入式改用户 MD** · LLM 改写一律走「生成 prompt 让用户跑 Claude」路径 · MD 永远是 SSOT
- **CHANGELOG / SemVer 偏移**:任何 SKILL.md 变更都是 minor · 这次同时改 `docs-cockpit-author` / `docs-cockpit` 两个 skill + build.py · 必须 0.11.0 minor

## §4 · Premises (locked in Phase 3)

1. **方向 premise** · v0.11+ 不是更精致的看板 · 是从「可视化看板」升级到「AI 协作驾驶舱」
2. **Schema 演进 premise** · `subtasks` 从字符串 list 升级为对象数组 · 字段至少 `id / title / status / code? / docs?` · **minor · 完全向后兼容旧 string 格式**(validator 把 string 当 `title` · 自动 normalize · 不报 error)
3. **Prompt scaffolding 边界 premise** · v1 面向人 copy-paste · 不直连 Claude · MCP / plugin hook 是 v2 范围
4. **Trust premise** · 「自动优化文档」不能侵入式改用户 MD · MVP 走「生成 prompt 让用户跑」· 保留 MD 是 SSOT 的信任锚
5. **Hard gate premise** · 本轮 office-hours 只设计 · 不实施 · 实施由后续 task 触发(/plan-eng-review 或直接动手)

## §5 · Approaches Considered

### Approach A · MVP 数据层 (W1 only)
**Summary**:仅升级 subtask 为一等公民 · 不做 prompt scaffolding · 不做 LLM 优化器
**Effort**:M (2-3 周)
**Risk**:Low

### Approach B · 数据层 + Prompt scaffolding (W1 + W3) ✅ CHOSEN

**Effort**:M-L (3-4 周)
**Risk**:Low-Med

### Approach C · 全功能 (W1 + W2 + W3)
**Summary**:B 之上加 `docs-cockpit suggest` 子命令(LLM 改写用户 MD 的 prompt 生成)
**Effort**:XL (4-6 周)
**Risk**:Med-High

## §6 · Recommended Approach (B · W1 + W3 详细设计)

### §6.1 · W1 数据层 · subtask 一等公民

**Frontmatter schema 演进**(`docs-cockpit-author` §2 更新):

```yaml
subtasks:
  # 旧格式 · 仍然支持 · validator 看到 string 当 title · 自动生成 id
  - "Lane A · BrowserVendor abstraction"

  # 新格式 · 推荐
  - id: M09-S1
    title: "Lane A · BrowserVendor abstraction"
    status: done                  # not-started | in-progress | done | blocked
    code: sourcery/worker/browser_vendor.py:42-89
    docs:
      - docs/spec/module/M09-browser-cluster.md#§2.1
      - docs/RFC/RFC-004-browser-vendor-abstraction.md

  # code 也支持 list (一个 subtask 跨多文件)
  - id: M09-S2
    title: "Lane B · LocalPlaywrightVendor"
    status: done
    code:
      - sourcery/worker/local_playwright_vendor.py
      - sourcery/worker/tests/test_local_vendor.py
    docs: [docs/spec/module/M09-browser-cluster.md#§2.2]
```

**`build.py::_build_card()` 改动**:
- 新增 `_normalize_subtasks(raw, module_id)`:把 string / dict 统一成 dict
  - string → `{id: auto-gen, title: string, status: not-started}`
  - **id 生成算法**:`<module-id>-<sha1(title)[:6]>` · **不用 body 顺序索引** · 否则用户在中间插入新 subtask 会让所有后续 id shift · 破坏 localStorage 持久化(localStorage 用 subtask.id 为 key)。Hash 算法的副作用:用户改 title 会换 id · 这是可接受 trade-off(改 title = 改任务定义 = 状态本来就该重算)
  - dict → 直接用 · `id` 缺失时按 hash 算法生成 · `status` 缺失默认 `not-started`
- subtask 拼装到 payload 时 · 解析 `code` ref · 调用新函数 `_resolve_code_anchor()`:
  - **`code` ref 格式决策(原 §7 Q1)** · 用 `path:start-end`(POSIX 风格 · grep 友好 · 跟 ripgrep / VS Code Quick Open 一致)· build 时同时生成 GitHub URL fallback(`<github-base>/blob/<branch>/<path>#L<start>-L<end>`)嵌入 payload · GitHub base 从 `docs-cockpit.yaml::repo_url` 读
  - 校验 `path` 存在(repo root 相对)
  - **路径处理统一 severity**:文件不存在 → **warn**(`severity: warn` · 不阻塞 build)· 这覆盖两种情况:用户笔误 + 后期 rename/delete · drawer 渲染该 subtask 时显示 `⚠ stale anchor` badge · 但 subtask 本身仍渲染 · status / docs / prompt 仍可用
  - 文件存在但行号越界 → **warn** · 行号显示 + badge
  - 路径存在 + 行号合法 → 读对应行 ±5 行 · 嵌入 `code_preview` 字段(长度 hard cap 800 字符 · 超出截断 + `…` 标记)
  - **Defensive IO error handling(plan-eng-review 3A)**:任何下列 IO 异常一律 **warn + None preview**(build 不炸)· drawer 显示 `⚠ preview unavailable` badge:
    - `OSError` / `PermissionError`(文件被锁 / 权限不足)
    - `UnicodeDecodeError`(文件不是 utf-8 · 例如 GBK / latin-1)
    - 文件前 1024 字节含 null byte → 判定 binary · skip
    - 文件 > 5MB → skip(超大文件 build 卡顿)
    - 行号越界(start > 文件行数)→ skip
  - **Performance(plan-eng-review 4A)**:`paths.py::_read_code_lines(path, start, end)` 用 `@functools.lru_cache(maxsize=256)` 包装 · 多 subtask 引用同 `path:lines` 不重复读 · build wall-time 节省 1.5-3x

**`validate_meta()` 改动**:
- 新增 §2.4 subtask schema 校验:
  - `id` 唯一性(单 module 内)· error 级别(同 id 会破坏 localStorage 状态)
  - `status` enum · error 级别(不在枚举内是输入错误)
  - `code` 路径存在性 + 行号合法性 → **warn** 级别(见上)
  - `docs` 路径走现有 `_resolve_doc_path()` 三步 fallback · path 不存在 warn
- `Issue.reference` 指向 `docs-cockpit-author · §2.4`

**`## 3 · 待办` body fallback 增强**(`build.py::extract_subtasks_from_body()`):

新内联语法:

```markdown
## 3 · 待办

- [x] Lane A · BrowserVendor abstraction @code:sourcery/worker/browser_vendor.py:42-89 @docs:M09-S1
- [x] Lane B · LocalPlaywrightVendor @code:sourcery/worker/local_playwright_vendor.py
- [ ] Lane F · Account/Proxy vendor_refs @code:sourcery/worker/account.py
```

- `[x]` / `[ ]` → status (done / not-started)
- `@code:<path>:<lines>` → subtask.code(支持多个)
- `@docs:<ref>` → subtask.docs(支持多个)
- id 自动生成 · **跟 frontmatter normalize 统一用 `<module-id>-<sha1(title)[:6]>`** · 不再用 body 顺序索引 · 这样混源 module(部分 subtask 在 frontmatter 部分在 body)id 计算一致 · 不冲突不漂移

**HTML drawer 渲染**(`templates/index.html.tmpl`):

```
[Subtasks · 5/8 done]
☑ M09-S1 · Lane A · BrowserVendor abstraction
   📁 sourcery/worker/browser_vendor.py:42-89  [open in VS Code]
   📄 M09-spec §2.1 · RFC-004
☑ M09-S2 · Lane B · LocalPlaywrightVendor
   📁 sourcery/worker/local_playwright_vendor.py
☐ M09-S6 · Lane F · Account/Proxy vendor_refs    [Copy prompt]
   📁 sourcery/worker/account.py
```

- `status` select 改 localStorage 用 `subtask.id` 为 key(不再用 index · 跨 build 稳定)
- `code` 图标点击:
  - 用户在 VS Code 打开 cockpit · 触发 `vscode://file/...` 深链
  - 用户用浏览器打开 · fallback 复制 path 到剪贴板 + toast
- `docs` 图标:点开打开关联 doc drawer(已有机制 · 直接复用)
- 每个 in-progress subtask 旁边显示 `[Copy prompt]` 按钮(进 §6.2)

### §6.2 · W3 Prompt scaffolding

**新模块** `docs_cockpit/prompt.py`:
- 依赖 Jinja2 · 加到 `pyproject.toml` `[project] dependencies`
- **Sandbox 必选**:用 `jinja2.sandbox.SandboxedEnvironment`(不是 default `Environment`)· 防止用户 template 用 `{% include %}` 任意读 fs · docs-cockpit 是 MIT 工具会在下游 untrusted repo 上跑 build · 安全边界重要
- **模板搜索路径白名单**:Loader 只允许两个目录 — `docs/prompts/`(用户)和 `docs_cockpit/templates/prompts/`(内置)· 用 `ChoiceLoader([FileSystemLoader('docs/prompts'), PackageLoader('docs_cockpit', 'templates/prompts')])`
- 函数 `render_prompt(module, subtask, repo_root, template_path?) -> str`
- Context vars 注入:
  - `module`:module 全 frontmatter + body
  - `subtask`:当前 subtask dict + code_preview(W1 已读)
  - `linked_docs`:`subtask.docs` 解析后的 doc 对象 list · **每个 doc 的 body summary hard cap 2000 字符**(取 §1 + §2 + §3 头部 · 超出截断 + `…` 标记 + 在 prompt 末尾加「完整 doc 见 `<path>`」hint)· 因为 prompt embed 走 sidecar(见 §6.3)· 这个 cap 控制 sidecar 单 entry 体积
  - `repo_root`:绝对路径
  - `current_branch`:**lazy + try/except** · `try: subprocess.check_output(['git', 'rev-parse', '--abbrev-ref', 'HEAD'], stderr=DEVNULL).decode().strip() except: None` · CI shallow-clone / 从 tarball build / 非 git 场景 fallback 为 `None` · template 用 `{% if current_branch %}...{% endif %}` 守护

**Context vars stability contract(plan-eng-review 2A)**:这是开源工具 · 下游 Sourcery + bastion + 第三方用户会写自定义 template · docs-cockpit 升级时不能静默 break:

1. **No-remove / no-rename**:每个 minor release 不删除 / 不重命名现有 vars · 只能 deprecate(打 warning · 至少保留两个 minor 周期再删)· 大版本(major)可以删 · 需 CHANGELOG 显式列
2. **New vars 必须可选**:新 var 引入时 · 内置 template 必须用 `{{ new_var | default('') }}` 或 `{% if new_var %}...{% endif %}` 守护 · 这样老用户 override template 不引用新 var 也能跑
3. **SandboxedEnvironment 配 `Undefined` (不是 `StrictUndefined`)**:用户 template 引用未定义 var 不 raise · 而是渲染为空 string · 优先 graceful degrade
4. **Schema 写到 `docs-cockpit-author/SKILL.md` §10.2**:列出 v0.X 提供的所有 vars · 每个 var 标 since-version · 每次 minor release 同步更新

这条契约写到 `references/prompt_templates.md` · 用户可以验证 · 升级有把握。

**Template 寻找顺序**(`prompt.py::_resolve_template`):
1. subtask frontmatter `prompt:` 显式指定(`docs/prompts/<custom>.md.j2`)
2. module frontmatter `prompt:` 显式指定
3. module frontmatter `prompt_kind:` 枚举(`feature` / `fix` / `refactor`)→ 用 docs-cockpit package 内置 template
4. fallback:`tpl/generic.md.j2`(docs-cockpit 内置)

**内置 templates**(`docs_cockpit/templates/prompts/`):
- `generic.md.j2` · 通用
- `feature.md.j2` · 实现 feature 子任务
- `fix.md.j2` · 修 bug 子任务
- `refactor.md.j2` · 重构子任务

每个 template 结构(generic 示例):

```jinja
你正在 docs-cockpit 驱动的 sprint **{{ module.sprint }}** 中执行子任务 `{{ subtask.id }}`。

## 上下文 · Module
- **Module**: {{ module.id }} · {{ module.title }}
- **目录**: {{ module.code_dir or 'N/A' }}
- **Status**: {{ module.status }} · Progress: {{ module.progress }}%
- **Spec**:
{{ module.body_summary }}

- **Status**: {{ subtask.status }}
{% if subtask.code %}
- **代码定位**:
{% for ref in subtask.code %}
  - `{{ ref.path }}`{% if ref.lines %}:{{ ref.lines }}{% endif %}
{% if ref.preview %}
    ```
    {{ ref.preview }}
    ```
{% endif %}
{% endfor %}
{% endif %}

{% if linked_docs %}
## 上下文 · 关联文档
{% for doc in linked_docs %}

{{ doc.summary }}
{% endfor %}
{% endif %}

完成后,请输出 frontmatter patch(只输出变更字段),格式:

```yaml

**CLI 子命令** · `docs-cockpit prompt`:
```bash
docs-cockpit prompt M09 M09-S6              # 子任务 prompt 输出到 stdout
docs-cockpit prompt M09 M09-S6 --copy       # 同时复制到剪贴板(pyperclip)
docs-cockpit prompt M09                     # 整 module 级 prompt
docs-cockpit prompt --list                  # 列所有可用 template
```

**HTML drawer 「Copy prompt」按钮**:
- build 时为每个 subtask 渲染好 prompt · **不内嵌主 HTML · 走 sidecar(详见 §6.3)**
- 浏览器点击 →
  1. `fetch('prompts.json').then(d => d['<subtask.id>'])` 拿渲染好的字符串
  2. 复制策略两层 fallback:
     ```js
     async function copy(text) {
       try { await navigator.clipboard.writeText(text); return true; }
       catch (e) {
         const ta = document.createElement('textarea');
         ta.value = text; ta.style.position='fixed'; ta.style.opacity='0';
         document.body.appendChild(ta); ta.select();
         const ok = document.execCommand('copy');
         document.body.removeChild(ta); return ok;
       }
     }
     ```
- **`file://` 兼容性**:`navigator.clipboard` 在 `file://` origin 大部分浏览器禁用 · cockpit 默认 file 协议打开 · 所以 `execCommand('copy')` fallback 是 primary path · 即使 deprecated 也必须保留
- toast 提示 success / failure(失败时 toast 显示 prompt 字符串前 200 字符 + 「请手动复制」)

**Lint 扩展** · `docs-cockpit lint --prompts`:
- 校验所有 `prompt:` 字段引用的 template 文件存在
- 校验 Jinja2 syntax · 校验引用的 context vars 在 schema 内

### §6.3 · HTML 嵌入策略 · sidecar + script-tag

**问题**(reviewer 指出):rendered prompt + code_preview + linked_docs summary 全部 inline 到 `state.json` payload · Sourcery 24 modules × 8 subtasks × ~(2KB prompt + 1KB preview + 2×2KB docs) ≈ 2-2.5MB inline JSON · `__DOCS_JSON__` 单 token 替换会膨胀 HTML 到 3MB · 跟 §8 「+30% 体积」目标冲突 · 浏览器解析慢 · grep diff 难看。

并且:任何 user MD / template / code preview 含字面串 `__DOCS_JSON__`(本 dogfood 项目里讨论 docs-cockpit 自己时极容易出现)都会破坏单 token 替换。

**决策**:

1. **主 payload 嵌入用 `<script type="application/json" id="docs-data">…</script>` · 不再用 `__DOCS_JSON__` 字符串替换**。
   - `templates/index.html.tmpl` 把 `__DOCS_JSON__` 占位符放到 `<script type="application/json" id="docs-data">__DOCS_JSON__</script>` 块内 · JSON 体内任何字面串(包括 `__DOCS_JSON__` 出现)都不会触发二次替换 · 因为 `template.replace()` 在 Python 端只跑一次
   - JS 读取:`const data = JSON.parse(document.getElementById('docs-data').textContent)`
   - 这同时解决了 collision · build.py 不需要换 sentinel(`__DOCS_JSON__` 仍然是 sentinel · 但只在 template 那一个位置存在 · `_render_html()` 用 `replace()` 仍然只 replace 一次)
   - 兜底:build 时 `escape` payload 里的 `</script>` 串(把 `</` 换成 `<\/`)· 防止 JSON 内出现 `</script>` 提前关闭 script tag

2. **prompt rendered string 拆到 sidecar `docs/prompts.json`**:
   - build 时单独输出 `docs/prompts.json`:`{"<subtask-id>": "<rendered prompt string>", ...}`
   - 主 HTML 不内嵌 prompt body · 只内嵌 subtask metadata(id / title / status / code / docs / has_prompt)
   - drawer 「Copy prompt」点击 → `fetch('prompts.json')` · 浏览器缓存友好 · 主 HTML 体积稳定
   - **file:// 限制**:`fetch('prompts.json')` 在 `file://` 部分浏览器(Chrome strict 模式)阻塞 · 兜底:build 时同时输出 `docs/prompts.js`(`window.__PROMPTS__ = {...}`)· HTML 用 `<script src="prompts.js" defer></script>` 引入 · 然后 drawer 直接读 `window.__PROMPTS__[id]` · 不走 fetch
   - 这个 JS sidecar 也走 sandbox safe content(只是 string map · 无 eval)

3. **code_preview** 走同一 sidecar `docs/code_previews.json`(或合并到 `prompts.js` 里 `window.__CODE__`)· 同上 file:// fallback

**体积预算**(决策后):
- 主 HTML(template + module/concept metadata + sprint/system_docs)· Sourcery 估算 ~600KB · 跟 0.10 基本持平
- `prompts.js`(24 × 8 × ~2KB)· ~400KB
- `code_previews.js`(24 × 8 × ~0.8KB)· ~160KB
- 总产物体积 ~1.2MB · 但主 HTML 单文件可独立打开 · drawer 「Copy prompt」点击才触发加载 prompts.js(via dynamic `<script>` inject 兜底 file:// fetch 限制)

**Trade-off**:用户不能再「打开 index.html · 完全断网工作」· 必须保留同目录的 `prompts.js` / `code_previews.js`。这是可接受的(本来就是 build 产物一起 ship)· 但要在 CHANGELOG + README 主推 banner 明确说明 · 0.10 的单文件叙事是 lead 卖点 · 不能闷头改。

**Graceful degrade(must)**:`prompts.js` 缺失时主 HTML 不能炸 · drawer 「Copy prompt」按钮 grey out + tooltip 「prompts.js 未加载 · 请确认 build 产物完整」· dashboard 其余功能(metadata / status / docs drawer)正常工作。

**Race condition handling**:`prompts.js` 用 `<script src="prompts.js" defer>` 在 `</body>` 前引入 · drawer 「Copy prompt」按钮 click handler 第一行 `if (!window.__PROMPTS__) { toast('prompts.js still loading, retry in a moment'); return; }` · 不假设加载顺序 · 不阻塞 dashboard 初渲染。

**Module 默认 lazy load**:如果用户 cockpit 实例很大(>50 modules)· `prompts.js` 也可能 >1MB · 留 v0.12 实现 per-module sidecar (`prompts/<module-id>.json` shard)· v0.11 单 sidecar 够用。

### §6.4 · 数据迁移 · 向后兼容

**Migrate path**:
- 已有 `subtasks: list[str]` 的 MD · build 时自动 normalize · 旧文件不需要改 · 但 cockpit 显示「⚠ 这个 subtask 缺 id/code/docs · run `docs-cockpit migrate-subtasks <file>` 升级」
- 新增 CLI `docs-cockpit migrate-subtasks <file>`:
  - 读 frontmatter + `## 3 · 待办` body
  - 对每个 string subtask · 用 LLM-free 启发(grep module 目录下 file/symbol)生成候选 `code:` ref
  - dry-run 默认 · 输出 diff · 用户 review 后 `--apply` 落盘
- 不强制迁移 · 老用户继续 v0.10 体验工作
