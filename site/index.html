为什么 AI 时代需要它

md 文件失控的真正原因，不是 AI 写太多。

是大模型每次写文档的字段、命名、格式都不一样——没有 schema 约束，AI 是最不可靠的"同事"。 你需要的不是更多管理工具，是一份规范让 AI 强制对齐。

▍ vibe coding 之前 每次问 AI "怎么写这个 plan"，它都给你一套新格式。
一个项目里 plan-01.md 用 status: 进行中，plan-02.md 用 state: in-progress，plan-03.md 干脆没有状态字段。
▍ docs-cockpit 之后 AI 写任何文档前，先读一份 docs-cockpit-author skill——里面规定了所有必填字段、命名规则、状态枚举。 AI 不再发明规则，它执行规则。
▍ vibe coding 之前 AI 把 progress 写成 60、status 写成 done——内部矛盾你肉眼根本看不出来。
上线那天才发现"已完成"的模块进度只有 60%。
▍ docs-cockpit 之后 每次 build 跑 validate_meta()，状态 × 进度的不变式不通过直接报错：
❌ M07.md · status:done 与 progress:60 矛盾
▍ vibe coding 之前 docs 目录里 47 个 md 文件，没人能一眼看出项目做到哪了。
想看进度只能挨个打开读，读完还是讲不清。
▍ docs-cockpit 之后 一行命令 docs-cockpit build，整个目录变成一个 HTML：
Kanban 看板 + Sprint 时间线 + 进度条 + 子任务清单 + 关联文档预览——本地双击就开。
▍ vibe coding 之前 Jira / Notion / Linear 都要复制粘贴一遍内容，数据托管在别人那。
想加新字段？等 SaaS 排期吧。
▍ docs-cockpit 之后 数据就在你 git 仓库的 markdown 里。
单 HTML 文件、零依赖、不联网、MIT 协议——所有权完全归你。

核心能力

给大模型一个能落地的项目规范。

v0.11 把 docs-cockpit 从"项目状态可视化"升级到"AI 协作驾驶舱"。每个能力都对应一个 vibe coding 实际会撞墙的场景。

▍

Frontmatter Schema 强校验

必填字段缺失、status×progress 矛盾、未知字段——全部在 build 阶段拦截。错误信息指向 author skill 第几条，AI 看到错误能自己改。

▍

AI 写文档前先读 skill

4 个 skill 覆盖 4 个场景：setup / 写文档 / 出周报 / 查状态。skill 是规范的载体——AI 不用每次问你"frontmatter 应该咋写"。

▍

Subtask 一键复制 prompt

每个子任务渲染好的 prompt（含上下文+代码锚点+关联文档），点 Copy 粘贴到 Claude Code / Cursor / Codex 就能让 AI 直接执行。

▍

Refine with AI · 双模式

"让 AI 帮我审一遍代码锚点对不对"——CLI agent 直接落 patch；浏览器 LLM 输出 patch 给你复制。一键交回，工程师在场内。

▍

多项目周报聚合

~/.docs-cockpit/ 注册所有项目，每周快照一次，AI 自动生成跨项目周报+周环比 diff（新完成/新阻塞/进度跃升/新增模块）。

▍

单文件 HTML · file:// 即开

不要服务器、不要 SSG、不要 JS 框架、运行时不发任何网络请求。生成的 dashboard 提交到 git，团队任何人 clone 下来双击就看到。

30 秒上手

三种装法，选你顺手的。

底层是同一个 Python CLI——Claude Code 用户一行命令搞定；其他 AI 编程工具用户用 pip / uv / pipx 装 CLI 即可。

▍ Claude Code 用户（推荐）

在 Claude Code 里直接装插件，跟它说"帮我把 docs 做成 dashboard"，它自己处理 Python 装机。

# 在 Claude Code 里 /plugin marketplace add Guohao1020/docs-cockpit /plugin install docs-cockpit@docs-cockpit # 然后跟 AI 说 "帮我把 docs 文件夹做成 dashboard"

▍ Cursor / Codex / 其他 AI 编辑器

手动装 Python CLI——uv / pipx / pip 三选一，然后初始化配置文件。

# 装 CLI（选一种） uv tool install docs-cockpit pipx install docs-cockpit pip install --user docs-cockpit # 在你项目根目录 docs-cockpit init docs-cockpit build

▍ Codex CLI / Codex App / Factory Droid / Gemini CLI / OpenCode / GitHub Copilot CLI 的原生打包在路线图上，schema/validator/spec 都是 agent-agnostic 的。

常见问题

你大概率想问的。

如果你的问题不在下面，欢迎到 GitHub Issues 留言。

为什么 AI 写出来的 markdown 越多反而越乱？
因为大模型每次写文档的字段、命名、格式都不一样——没有 schema 约束，AI 是最不可靠的"同事"。你今天问它"plan 该有哪些字段"它给一套答案，明天再问换一套。docs-cockpit 把规范写进一份 skill，AI 每次写 plan / RFC / spec 都先读规范，落盘前还会跑 frontmatter 验证，错了立刻指向规范第几条。
docs-cockpit 和 Notion / Jira / Linear 有什么区别？
Notion / Jira / Linear 是 SaaS——你要复制粘贴内容过去，数据在他们手里，多人用还得交月费。docs-cockpit 直接读你 git 仓库里的 markdown，零依赖、不联网、单 HTML 文件双击就开。AI 改 md，重跑一行命令，看板就同步。哲学是"markdown 是 source of truth，dashboard 只是一种视图"。
我不是程序员，可以用吗？
可以。如果你已经在用 Claude Code / Cursor 写项目笔记、整理资料、做产品文档，docs-cockpit 就是给你的。最低门槛：装一次 Claude Code 插件（一行命令），然后跟 AI 说"帮我把 docs 文件夹做成看板"，它自己装 Python、自己跑、自己出图。你只需要看结果。
支持哪些 AI 编程工具？
原生支持 Claude Code（插件市场一行命令安装）。底层是 Python CLI + markdown skill，所以 Cursor / Codex CLI / Codex App / Factory Droid / Gemini CLI / OpenCode / GitHub Copilot CLI 用户可以手动装 CLI 后使用——所有 agent 共用同一份 schema 规范。原生打包正在路线图上。
数据会上传吗？要付费吗？
完全免费、MIT 协议开源、运行时零网络请求。dashboard 是一个本地 HTML 文件，你可以双击打开，也可以提交到 git 仓库作为团队共享。所有数据都在你自己的硬盘和 git 仓库里。
大模型生成的 frontmatter 经常出错怎么办？
这是 docs-cockpit 专门解决的问题。每次 build 都会跑 validate_meta()，输出三级问题：
▍ error（看板没法渲染这个文档，比如 id 缺失）
▍ warn（能渲染但状态可疑，比如 status:done 但 progress:60）
▍ hint（可润色，比如缺 desc 会让"复制 prompt"上下文不全）
每条问题都带 💡 fix 修复建议和 📚 see 规范引用——AI 看到错误能自己改。
已有的 docs/plans/ docs/RFC/ 目录怎么迁移？
跟你的 AI 说"用 docs-cockpit 接管这个项目"。docs-cockpit skill 会触发 migrate 流程：先 dry-run 给你看会动哪些文件，确认后 git mv 重命名 + 注入 frontmatter scaffold，全程在 git 历史里可追溯。
能集成到 CI / pre-commit 吗？
可以。docs-cockpit lint 输出 stdout 三段式（error/warn/hint），且 --strict 模式会因 error 退出非零——直接接到 GitHub Actions / pre-commit hook 都行。state.json 是稳定的机器可读 sidecar（0.2.0+ 只加字段不减字段），外部工具读它做不变式检查。 v0.11 仓库本身就有 9-cell CI 矩阵（3 Python × 3 OS）+ 118 单测，可以直接参考。
